# NOC Living Docs 和相邻工具的边界

NOC Living Docs 不试图做完整知识库，也不替代团队已经在用的 README、ADR、issue tracker 或自动文档。它只负责把和代码长期同步最容易失真的一组信息放在固定位置：需求、当前行为、限制、测试记录和重要变更。

## NOC 负责什么

- 让每个 feature 有稳定文档目录。
- 让代码路径可以映射到 feature。
- 在提交和 CI 中提示“代码改了，相关 NOC 文档是否也该同步”。
- 帮助人和编程助手先读最小必要文档，而不是重新解释整个项目。

## NOC 不替代什么

| 概念 | NOC 不替代的部分 | NOC 负责的部分 |
|---|---|---|
| Wiki | 团队制度、会议记录、宽泛知识库、跨项目资料。 | 和本仓库 feature 行为直接绑定的长期维护文档。 |
| ADR | 架构决策的背景、选项、取舍和不可逆决策记录。 | 决策落到具体 feature 后的需求、限制、现状和测试影响。 |
| README | 项目简介、安装、运行、开发入口、常见命令。 | README 链接到 NOC；NOC 保存更细的 feature 维护语境。 |
| docs-as-code | 任意技术文档用 Git 管理的做法。 | 一套固定目录和检查规则，专注代码变更和 feature 文档同步。 |
| MCP code index | 给工具检索代码结构、符号或语义索引。 | 给人和工具定位“这段代码属于哪个业务 feature、应读哪些文档”。 |
| 自动生成代码文档 | 从代码生成 API、类型、函数或注释说明。 | 记录代码无法可靠生成的意图、边界、测试结论和人工确认的变更原因。 |
| 项目管理工具 / issue tracker | 任务分配、排期、状态流转、讨论和验收流程。 | 合并后仍应留在仓库里的 feature 行为事实和维护记录。 |

## 适合放在哪里

| 信息 | 推荐位置 | 判断规则 |
|---|---|---|
| 项目是什么、怎么启动、怎么测试 | `README.md` | 新人第一次打开仓库就需要看到。 |
| 某个功能应该做什么 | `noc_docs/.../requirements.md` | 会影响实现和验收，且需要随代码长期维护。 |
| 某个功能当前实际如何工作 | `noc_docs/.../status.md` | 描述当前代码事实，不是未来计划。 |
| 不能破坏的限制、安全边界、兼容规则 | `noc_docs/.../guardrails.md` | 改代码前必须知道，否则容易引入回归。 |
| 测试命令、验证结果、未覆盖风险 | `noc_docs/.../test-record.md` | 和 feature 行为或风险直接相关。 |
| 重要修改原因和影响范围 | `noc_docs/.../change-record.md` | 未来维护者需要知道为什么这么改。 |
| 大型架构选择和替代方案 | ADR | 重点是决策过程，而不是单个 feature 的日常维护事实。 |
| 临时任务、负责人、截止时间 | Issue tracker | 任务生命周期结束后不一定需要留在仓库。 |
| 团队制度、会议纪要、跨项目背景 | Wiki | 不直接绑定本仓库代码路径。 |
| API 参考、类型签名、函数说明 | 自动生成文档或 API docs | 可以从代码或注释稳定生成。 |
| 代码符号、调用关系、检索索引 | MCP code index | 给工具做检索，不承载人工维护的产品意图。 |

## 和 README 的关系

README 是入口，NOC 是维护面。README 应该告诉读者项目怎么运行、哪里找 NOC 文档、团队如何执行检查；NOC 文档不应该复制 README 的启动说明，而应该记录具体 feature 的需求、现状、限制、测试和变更。

## 和 ADR 的关系

ADR 记录“为什么选择这个架构”。NOC 记录“这个选择对某个 feature 的当前行为和维护边界意味着什么”。如果一个架构决策影响多个 feature，可以在 ADR 中写决策，在相关 `guardrails.md` 或 `status.md` 中写落地约束，并互相链接。
